# v0.5.0 — Signed manifests

Tracking issue for the next release line. Mirrors the GitHub issue
of the same name
([#31](https://github.com/marmar9615-cloud/agentbridge-protocol/issues/31));
kept in-tree so design docs and PRs can link to a stable path
regardless of issue renumbering.

## Goal

Design — and later implement — **signed AgentBridge manifests** so
agents can verify a manifest's authenticity and integrity before
trusting any action it declares. Designing this carefully *before*
writing any signing code keeps v1.0 readiness on track and avoids
shipping a verification path we can't change later.

## Why

Web apps publish machine-readable manifests at
`/.well-known/agentbridge.json`. Agents read the manifest and call
the actions it declares. Today, the only thing standing between a
malicious manifest and an agent is **transport security plus origin
pinning**:

- TLS proves the bytes came from `app.example.com` over a secure
  channel.
- Origin pinning ([`assertSameOrigin`](../../apps/mcp-server/src/safety.ts))
  ensures action endpoints stay on the publisher's origin.

That leaves several real risks unaddressed:

- A compromised CDN cache or a stolen TLS cert lets an attacker
  serve a substituted manifest with no protocol-layer signal.
- A stale manifest (replayed long after rotation) is
  indistinguishable from a fresh one.
- An adopter operating across multiple environments has no way to
  verify "this manifest is the one our build pipeline produced."

Signed manifests close those gaps by binding a manifest to a
publisher key the agent can verify offline.

## Security requirements (non-negotiable)

- **No private keys in manifests, ever.** The manifest carries the
  signature; the signing key stays out of the published surface.
- **No secrets in docs or examples.** All sample keys are clearly
  labeled non-production, regenerable from the doc.
- **Key rotation must be a first-class concern.** The design must
  specify how a publisher rotates keys without breaking in-flight
  agent sessions, and how revocation is propagated.
- **Verification failure modes must be explicit.** Missing
  signature, invalid signature, expired manifest, unknown key id,
  malformed public key, base-URL/fetch-origin mismatch — each case
  has a documented behavior.
- **Replay / staleness must be addressed.** Stale signed manifests
  must be detectable. The design must specify a freshness model
  (timestamp + max-age, or equivalent).
- **Existing safety controls must be preserved.** Verification
  cannot bypass the confirmation gate, origin pinning, target-origin
  allowlist, audit redaction, stdio stdout hygiene, or the HTTP
  transport's auth/origin checks. Verification is **additive**.

## Non-goals

- **No full implementation in this PR.** Only design, ADR, and the
  cross-referenced doc updates.
- **No central certificate authority** in the first design unless
  explicitly justified. AgentBridge stays self-hosted.
- **No blockchain requirement.** Verification must work offline
  without a third-party network call.
- **No removal of unsigned-manifest support yet.** v0.5.0 ships
  signing as opt-in; unsigned manifests continue to work, with a
  scanner downgrade.
- **No package version bump** in the design PR. Implementation will
  bump versions when it lands.

## Acceptance for v0.5.0 release (later)

- Manifest signature envelope implemented per the chosen design.
- Canonical-JSON serializer shipped from `core` with cross-language
  test vectors.
- SDK exposes `signManifest()` / `createSignedManifest()` and a
  `verifyManifestSignature()` API.
- Scanner reports unsigned manifests as an `info`/`warning`-level
  downgrade and reports invalid signatures as `error`.
- MCP server verifies signatures when present and refuses verified-
  invalid manifests; unsigned manifests still work in v0.5.0 mode
  (opt-in `--require-signature` for high-assurance deployments).
- CLI gains `agentbridge sign`, `agentbridge verify`, and
  `agentbridge keys (generate|list|rotate)` per the design.
- Threat model T1 is downgraded from "open" to "mitigated for
  signed manifests" with documented assumptions.
- Tests for every requirement in the design's testing plan.
- Release notes flag signing as opt-in and document new env vars
  and CLI commands.
- Trusted Publishing flow exercised end-to-end for v0.5.0.
