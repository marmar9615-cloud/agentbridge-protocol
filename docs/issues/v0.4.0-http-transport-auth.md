# v0.4.0 — HTTP MCP transport + auth

Tracking issue for the next release line. Mirrors the GitHub issue
of the same name; kept in-tree so design docs and PRs can link to a
stable path regardless of issue renumbering.

## Goal

Add an opt-in **Streamable HTTP** MCP transport to the AgentBridge
MCP server while preserving stdio as the default. Hosted and remote
MCP clients need an HTTP option; stdio remains the right default for
local desktop clients (Codex, Claude Desktop, Cursor, custom).

## Why

- The MCP specification defines two transports — **stdio** and
  **Streamable HTTP**. AgentBridge currently ships only stdio.
- Hosted clients and centralized agent platforms cannot launch a
  local subprocess; they need an authenticated HTTP endpoint.
- Local desktop clients should keep using stdio. It is cheaper to
  launch, has no listening socket, and reads credentials from
  environment variables instead of doing an HTTP auth dance.
- Shipping HTTP without authentication, Origin validation, or a
  loopback default would invert the safety story we built in
  v0.3.0. Designing this carefully *before* writing transport code
  keeps the v1.0 bar from slipping.

## Security requirements (non-negotiable)

- **Authentication required.** No unauthenticated remote HTTP. No
  query-string tokens. Bearer tokens in the `Authorization` header
  only.
- **Origin validation.** Validate the `Origin` header on every
  request that includes one; reject unknown origins with HTTP 403.
- **Loopback bind by default.** Default to `127.0.0.1`; binding to
  `0.0.0.0` or a public interface requires explicit opt-in.
- **No query-string tokens.** Bearer tokens go in the
  `Authorization` header. Reject any token presented in the URL
  query string.
- **Explicit remote-bind opt-in.** Public bind without auth must
  fail hard. Public bind without an Origin allowlist must warn or
  fail depending on mode.
- **Audit caller identity.** Audit events must record `transport`
  and a non-secret caller identifier so HTTP calls can be
  distinguished from stdio calls.
- **Reuse existing safety code.** The same confirmation gate,
  origin pinning, target-origin allowlist, idempotency, and audit
  redaction must run regardless of transport. No parallel safety
  paths.

## Non-goals

- **No full OAuth 2.1 authorization server** in the first
  implementation. Phase 1 is a static bearer token. The design must
  leave room for an OAuth resource-server mode later, but the
  initial implementation is the smaller, well-scoped piece.
- **No removal of stdio.** stdio remains the default and the
  primary path for local desktop clients.
- **No production-hosting claim yet.** Shipping HTTP transport does
  not declare AgentBridge production-ready. The v1.0 readiness
  checklist still applies.
- **No real destructive demo actions.** The demo app continues to
  simulate refunds, etc. HTTP transport changes nothing about the
  demo's safety story.

## Acceptance for v0.4.0 release (later)

- Streamable HTTP transport implemented behind an explicit flag /
  env var.
- Bearer-token auth enforced on every HTTP request.
- Origin allowlist enforced on every HTTP request that supplies
  `Origin`.
- Loopback bind by default; public bind requires opt-in.
- Audit events carry `transport` and caller metadata.
- Tests for every requirement in the design's testing plan.
- Release notes flag HTTP as opt-in and document the env vars.
- Trusted Publishing flow exercised end-to-end for v0.4.0.
