**Date:** 2026-04-26
**Status (historical):** Public beta — first cut intended for outside
developer evaluation. Not production security infrastructure.
**Distribution (historical):** **Source-only.** The
`@marmar9615-cloud/agentbridge-*` packages were *prepared* for npm
publishing but never published under that scope; they were renamed to
`@marmarlabs/agentbridge-*` for the v0.2.0 stable release.

AgentBridge is an **AI-native action layer for web apps.** Web apps
publish a machine-readable manifest at `/.well-known/agentbridge.json`
declaring their structured actions. AI agents discover and invoke those
actions through an MCP server that enforces safety guarantees
(confirmation gates, origin pinning, audit logging).

## What's in 0.2.0-beta

This release is the first that's packaged for outside developers to
clone, evaluate, and (once published to npm) consume as installed
packages. It is a packaging/release-prep release on top of the Phase 2
developer-tooling work shipped in v0.2.0.

### What changed since 0.2.0 (Phase 2)

- **Renamed** all publishable packages from `@agentbridge/*` to
  `@marmar9615-cloud/agentbridge-*`. The `@agentbridge` scope is
  unowned and unverifiable. See [docs/npm-scope.md](../npm-scope.md).
- **Real build pipeline.** `npm run build` produces `dist/index.js` +
  `dist/index.d.ts` (and `dist/bin.js` for the CLI) for every
  publishable package.
- **`npm run pack:dry-run`** validates each tarball.
- **`npm run smoke:external`** simulates an external clone.
- **Per-package READMEs** for every publishable package.
- **CI** typechecks all 6 packages, builds, and runs pack-check.
- **release-check workflow** for pre-tag validation.
- **Issue templates, PR template, Dependabot config**.

No runtime behavior changed; all 86 tests still pass.

## Install

### Today — clone the repo (source-only)

```bash
git clone https://github.com/marmar9615-cloud/agentbridge-protocol.git
cd agentbridge-protocol
npm install
npm test
npm run dev   # demo on :3000, Studio on :3001
```

### After npm publishing (separate manual step — not yet done)

Once the maintainer has run the publish flow in
[docs/npm-publishing.md](../npm-publishing.md), the equivalent install
becomes:

```bash
# Author manifests in your own app
npm install @marmar9615-cloud/agentbridge-sdk @marmar9615-cloud/agentbridge-core

# Run the CLI
npx @marmar9615-cloud/agentbridge-cli scan http://localhost:3000

# Run the MCP server
npx @marmar9615-cloud/agentbridge-mcp-server
```

## Try it

The snippets below all use the `npx @marmar9615-cloud/agentbridge-*`
form for what consumption will look like once the packages are on npm.
**Until npm publishing happens, run these from a local checkout** —
substitute `npm run dev:cli -- <args>` for `npx @marmar9615-cloud/agentbridge-cli <args>`,
and `npm run dev:mcp` for `npx @marmar9615-cloud/agentbridge-mcp-server`.

### CLI

```bash
# Post-publish form (works after npm publishing):
npx @marmar9615-cloud/agentbridge-cli scan http://localhost:3000
npx @marmar9615-cloud/agentbridge-cli validate ./public/.well-known/agentbridge.json
npx @marmar9615-cloud/agentbridge-cli generate openapi ./store.openapi.json --out ./agentbridge.json
npx @marmar9615-cloud/agentbridge-cli init

# Local-checkout form (works today, from inside this repo):
npm run dev:cli -- scan http://localhost:3000
npm run dev:cli -- validate ./public/.well-known/agentbridge.json
npm run dev:cli -- generate openapi ./store.openapi.json --out ./agentbridge.json
npm run dev:cli -- init
```

### OpenAPI import

Take an existing OpenAPI 3.x doc and turn it into a draft manifest:

```bash
npx @marmar9615-cloud/agentbridge-cli generate openapi ./your-api.openapi.json \
    --base-url https://api.example.com \
    --out ./public/.well-known/agentbridge.json
```

Risk is inferred from HTTP method (GET → low, POST/PUT/PATCH → medium,
DELETE → high). Always review before publishing.

### MCP setup

Edit `~/Library/Application Support/Claude/claude_desktop_config.json`
(macOS) or `%APPDATA%\Claude\claude_desktop_config.json` (Windows). For
**post-publish** consumption:

```json
{
  "mcpServers": {
    "agentbridge": {
      "command": "npx",
      "args": ["@marmar9615-cloud/agentbridge-mcp-server"],
      "env": { "AGENTBRIDGE_ALLOW_REMOTE": "false" }
    }
  }
}
```

For **today** (source-only, requires `npm run build` to produce
`dist/index.js`):

```json
{
  "mcpServers": {
    "agentbridge": {
      "command": "node",
      "args": [
        "/absolute/path/to/agentbridge-protocol/apps/mcp-server/dist/index.js"
      ],
      "env": { "AGENTBRIDGE_ALLOW_REMOTE": "false" }
    }
  }
}
```

Restart Claude Desktop. Five tools should appear: `discover_manifest`,
`scan_agent_readiness`, `list_actions`, `call_action`, `get_audit_log`.

## Known limitations (deliberately not in beta)

- **Signed manifests** — a consumer can't cryptographically verify
  the manifest publisher.
- **HTTP MCP transport** — stdio only.
- **OAuth scope enforcement** — `permissions[]` is documentary.
- **Distributed audit storage** — local JSON file only.
- **Policy engine** (OPA / Cedar) — not wired.
- **Rate limiting / cost accounting** — none.
- **Multi-tenant isolation** — none.

See [docs/public-beta.md](../public-beta.md) for the full list.

## Safety notes

The MCP server enforces:

- Risky actions require `confirmationApproved: true` AND a single-use,
  input-bound `confirmationToken`.
- Action endpoints must share origin with `manifest.baseUrl`.
- Loopback URLs only by default; `AGENTBRIDGE_ALLOW_REMOTE=true` is the
  only escape.
- `authorization`, `cookie`, `password`, `token`, `secret`, `api_key`
  keys are redacted from the audit log.
- Demo-app destructive actions are simulated; no real services are
  touched.

For security issues, see [SECURITY.md](../../SECURITY.md). Do not open
a public issue.
